Hermes Kanban Runtime Architecture

Dispatcher tick → worker fork → kanban task lifecycle. One task, one run, one worker process.

Dispatcher — Traffic Cop

  • • Singleton loop inside gateway
  • • Picks ready tasks every tick
  • • Forks workers, writes claim_lock
  • • Cleans expired claims, dead PIDs
  • • Enforces max_runtime_seconds
  • • Spawn-failure circuit breaker

Worker — The Executor

  • • OS process forked by dispatcher
  • • Runs agent CLI (claude / codex / …)
  • • Owns one workspace dir, one TASK_ID
  • • Streams stdout → gateway tail
  • • Must call kanban_complete or kanban_block
  • • Silent exit → protocol violation

Tasks + Runs — Persistent State

  • • ~/.hermes/kanban.db (SQLite)
  • • tasks row = work item (noun)
  • • task_runs row = one execution (audit)
  • • status, claim_lock, current_run_id on tasks
  • • terminal_protocol on task_runs
  • • Shared blackboard, no separate state svc

Dashboard + Workspace — UI

  • • Dashboard :9119 reads kanban.db
  • • SwitchUI :3000 proxies dashboard
  • • TaskDetailDrawer → RunsTab
  • • Surfaces every terminal_protocol value
  • • Red banner on protocol violation
  • • SSE events for live status updates

Failure Modes

  • • Silent rc=0 exit → re-claim next tick
  • • Hang past max_runtime → SIGTERM/SIGKILL
  • • PID dead but claim_lock held → reset (macOS zombie caveat)
  • • N consecutive spawn fails → circuit breaker parks task
  • • Bad column data (ISO in INTEGER col) → board fetch 503 upstream
